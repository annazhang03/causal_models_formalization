
# Causality and Semantic Separation

This repository contains the Rocq proof artifact for the paper _Causality and Semantic Separation_.
It formalizes causal models and mechanizes the proof that the paper's notion of _semantic separation_
is equivalent to _d-separation_, the standard graphical independence criterion used in
causal inference.

## Getting Started Guide
Run the following commands from the root of the artifact directory.
1. Install Docker: https://www.docker.com/products/docker-desktop/.
2. Build the Docker image using the provided `Dockerfile`. This installs dependencies and compiles the Rocq development: 
```bash
docker build -t semantic-separation .
```
3. Run the container:
```bash
docker run -it semantic-separation
```
4. Inspect the definition `semantically_separated` in `Semantics/SemanticSeparationDef.v`. This corresponds to
Definition 3.3 in the paper.
5. Inspect the theorem `semantic_and_d_separation_equivalent` in `Semantics/SemanticDSepEquiv.v`.
This corresponds to Theorem 3.7, the main result of the paper.
6. To inspect the assumptions of the main theorem, uncomment `Print Assumptions semantic_and_d_separation_equivalent.`
at the bottom of `Semantics/SemanticDSepEquiv.v` and run:
```bash
make
```
Rocq will report the assumptions used in the proof, which includes functional extensionality (`functional_extensionality_dep`).

## Step-by-Step Instructions

### 1. Claims Supported by the Artifact

The artifact mechanizes and verifies the main theoretical results of the paper. In particular,
it contains formal Rocq definitions and proofs (where applicable) of:
- The definition of semantic separation (Definition 3.3).
- The definition of _d_-separation.
- Supporting lemmas relating graph structure and semantic behavior.
- Theorem 3.7: equivalence between semantic separation and _d_-separation.

Compiling the development mechanically checks these proofs.
The table in Section 6 of this document maps each definition and theorem in the paper to its corresponding Rocq definition or theorem.

### 2. System and Build Information
- **Rocq version:** 9.1.1
- **OCaml version:** 5.4.0
- **Build system:** Makefile (auto-generated by `rocq makefile`)
- The `_CoqProject` file lists all the directories and source files necessary to build the development.

Building the Docker image already generates the Makefile and compiles the code.
To regenerate the Makefile and rebuild manually, run:
```bash
rocq makefile -f _CoqProject -o Makefile

```

### 3. Repository Structure
- `Utils`: foundational definitions and helper lemmas for lists, logic, and type theory. 
   This directory contains general-purpose utilities that do not depend on graphs or causal concepts.
- `DAGs`: formalization of directed acyclic graphs (DAGs) in Rocq. Includes definitions of nodes, edges,
   and paths, as well as key properties such as acyclicity, reachability, pathfinding, and topological
   sorting.
- `CausalDiagrams`: extensions of DAGs to causal settings. Defines notions specific to causal 
reasoning, such as _d_-separation, intermediate node classification, and causal paths, building directly on the `DAGs` layer.
- `Semantics`: development of the semantic perspective on causal models. Includes the function-based
framework for causal semantics, the notion of semantic separation, and all supporting lemmas and 
theorems culminating in the fully mechanized equivalence proof between semantic separation and
_d_-separation.

### 4. Assumptions and Admitted Lemmas
- The development assumes the standard logical principle of Functional Extensionality
(`functional_extensionality_dep`, imported from `Coq.Logic.FunctionalExtensionality`).
- Although Appendix B.1 of the paper notes that some graph-theoretic theorems were not fully proven in our formalization, these proofs have since been completed. Thus, all theorems on which the main equivalence theorem depends are now fully mechanized in Rocq.  
- A small number of additional theorems are Admitted in `CausalDiagrams/Interventions.v` and in the `Spirtes/` directory;
these results are auxiliary and not referenced in the semantic development or the paper.

### 5. Key Content

The central result of this artifact is a new semantic notion, semantic separation, together with a proof that it is equivalent
to _d_-separation. To support this result, the repository contains a number of intermediate definitions, lemmas, and theorems.

Section 6 of this document (duplicated from Appendix A of the paper) provides a complete mapping from definitions, lemmas, and theorems
in the paper to their corresponding Rocq identifiers. Section 6 also lists the locations of the main types defined in the paper.

The following subsections provide a more detailed guide to the most important concepts in the paper and describe where they appear in the repository.

#### 5.1. Functional Semantics
In Section 3 of the paper, we introduce the function-based formal semantics using primarily mathematical notation. In the paper, assignments (e.g. unobserved-terms assignments $U$) are represented as a function $U: \mathcal{V} \rightarrow X$. In the repository, however, $U$ has type `assignments`, and the value $U(w)$ is accessed using `get_assigned_value U w`, defined in `CausalDiagrams/Assignments.v`.

Similarly, in the paper we write $f_U: \mathcal{V} \rightarrow X$ for `nodefun` $f$ evaluated with unobserved-terms assignments $U$. In the repository this value is obtained with the function `find_value`, defined in `Semantics/FindValue.v`, with $U$ supplied as one of the arguments.

#### 5.2. Semantic Separation Definition
The definition of semantic separation appears as Definition 3.3 in the paper and as the proposition `semantically_separated` in `Semantics/SemanticSeparationDef.v` in the codebase. 

Definition 3.3 is presented in the paper with five conditions; the Rocq definition follows the same structure.

**Condition 1 (Lines 275-277)**
- The requirement that $f_{U_\alpha}(u) = \alpha$ is ensured by 
`get_values G g Ua [] = Some Aa /\ unobs_valid_given_u G Ua Aa u a`.
- The predicate `unobs_valid_given_u` guarantees that $U_\alpha$ (`Ua`) is a valid assignment for the nodes of $G$ and that $g_{U_\alpha}(u) = \alpha$.
The structure `Aa` stores the resulting assignments for all nodes in $G$.
- The condition that $f_{U_\alpha}$ properly conditions on $Z$ is expressed by `unobs_conditions_on_Z G g Ua AZ Z`, which ensures that each node in $Z$ evaluates (via $g$) to its assigned value in `AZ`.

**Condition 2 (Lines 278-281)**
- The requirement that $f_{U_\beta}(u) = \beta$ is ensured by `get_values G g Ub [] = Some Ab /\ (unobs_valid_given_u G Ub Ab u b)`.
- The requirement that $U_\beta$ differs from $U_\alpha$ only for nodes in $\text{Anc}_Z^*(u)$ is expressed  by the proposition `assignments_change_only_for_subset`, defined in `CausalDiagrams/Assignments.v`. This ensures that both structures represent assignments for the same nodes, and that nodes outside the specified subset evaluate to the same value (or None) in both. 
- The function `find_unblocked_ancestors`, used to compute the relevant subset of nodes, is defined in `CausalDiagrams/UnblockedAncestors.v` and relies on a directed path-finding function.

**Condition 3 (Line 288)**
- The proposition `assignments_change_only_for_Z_anc_seq` captures this condition recursively. It again uses `assignments_change_only_for_subset`, now with a more complex subset.
- The function `unblocked_ancestors_that_changed_A_to_B` extracts nodes whose values differ between the two assignment sets.
- The function `find_unblocked_ancestors_in_Z` identifies unblocked ancestors of conditioned nodes that themselves have unblocked ancestors in this set. The condition then recurses over each triple of consecutive assignments.

**Condition 4 (Line 284)**
- `L` corresponds to the list $[U_1, ..., U_\ell]$, which is required to have length bounded by the number of nodes in $G$.

**Condition 5 (285-287)**
- The formalization of this condition is similar to that of Condition 1, with the main difference being that when $\ell = 0$, the condition applies to $U_\beta$. This is implemented by applying the condition to `last (Ub :: L) Ub`.

Finally, the requirement that the value of $v$ remains unchanged corresponds to Line 290:
`get_assigned_value Aa v = get_assigned_value Ab' v.`.

#### 5.3. Semantic Separation Equivalence Proof
The equivalence between semantic separation and _d_-separation is Theorem 3.7 in the paper. In the repository, it is mechanized in `Semantics/SemanticDSepEquiv.v` as the theorem `semantic_and_d_separation_equivalent`.

As described in the paper, the proof is decomposed into the two directions of the equivalence, each proved via contrapositive.

The precondition `generic_graph_and_type_properties_hold`, defined in `Semantics/FunctionRepresentation.v`, requires that the graph $G$ be acyclic, that the values type `X` contain at least two distinct elements, and that the graph be well-formed in Rocq (e.g., all nodes referenced in edges appear in the node set, and nodes and edges do not appear more than once). Similar well-formedness constraints are imposed on the set $Z$.

The theorem `semantic_and_d_separation_equivalent` corresponds directly to Theorem 3.7 of the paper. Uncommenting the line
```bash
Print Assumptions semantic_and_d_separation_equivalent.
```
will show that the only assumption is functional extensionality, as described in Section 4 above.

Throughout the paper we introduce several types; their corresponding Rocq definitions are listed below:
- `graph`,  `nodes`, and  `edges`  are defined in  `DAGs/Basics_Constr.v`.
